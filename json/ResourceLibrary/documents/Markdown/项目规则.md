# 一、文档管理规范（核心基础）

## 1.1 文档创建时机

新项目开发启动时，必须优先创建「说明文档.md」，作为项目全生命周期唯一管理载体。

## 1.2 文档核心内容

包含项目规划、实施方案、进度记录三大模块，内容需明确可落地（进度标注具体时间节点）。

## 1.3 文档更新要求

- 重新打开项目前，必须先阅读「说明文档.md」，确认进度与要求后再启动开发；

- 项目计划、进度节点变更时，即时更新文档对应模块；

- 完成一项任务（功能开发、BUG修复等），立即在文档进度记录中标记完成并补充结果说明。

# 二、开发流程规范（过程管控）

## 2.1 任务管理逻辑

采用<MCPsequential thinking>（顺序化思考）分析需求，拆解为可执行ToDoList，明确任务优先级与依赖关系。

## 2.2 任务执行闭环

严格按ToDoList顺序执行，完成一项后，需在文档及任务清单中标记“已完成”，确认无遗留问题再启动下一项，直至全量完成。

# 三、问题解决规范（技术支撑）

## 3.1 问题处理优先级

开发中遇技术问题，优先通过<MCPContext7>工具查找最新文档、代码示例，尝试自行解决。

## 3.2 官方文档依赖

若<MCPContext7>未解决问题，必须查找对应编程语言官方文档（如Python查Python.org、Java查Oracle官方文档），严禁自行编造代码或解决方案。

# 四、执行约束规范（硬性要求）

## 4.1 三大“绝不允许”原则

- 绝不允许项目延期：严格按文档时间节点执行，提前识别风险并调整计划（同步更新文档）；

- 绝不允许超出计划：开发范围、资源投入匹配文档规划，扩容需先更新规划并确认可行性；

- 绝不允许出错：开发中多轮自检，出现错误立即暂停，按问题解决规范处理，修复后验证无误再推进。

# 五、环境与输出规范（基础保障）

## 5.1 开发环境统一

固定使用Mac系统进行TRAE编程开发，保障环境一致性，避免跨系统兼容性问题。

## 5.2 代码输出标准

生成代码时，必须为所有函数添加函数级注释（包含功能描述、参数说明、返回值类型及用途）。
